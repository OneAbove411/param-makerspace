01 — Platform Overview

What Is This?

Param Makerspace is a community-driven digital platform for a physical makerspace. It connects makers, mentors, and administrators through a progression-based system where users earn XP, unlock ranks, complete challenges, build projects, attend events, earn badges, and purchase tools from the store.

Core Purpose

Structured Learning

Challenges in 3 tiers (Explorer → Solver → Architect) guide makers from beginner to advanced skills across 7 domains.

Core Purpose

Project Documentation

Makers document builds with images, videos, files, milestones, and team members. Projects go through a review pipeline (draft → pending → active).

Core Purpose

Community Events

Build challenges, maker meetups, and tech tuesday sessions — with team formation, submissions, and showcasing capabilities.

Core Purpose

Progression System

XP → Ranks → Badges → Feature Unlocks → Store Access. Every action contributes to a maker's growth path through 6 ranks.

03 — Visual Language

Design System

Aesthetic

Brutalist Industrial

Inspired by hardware documentation, technical manuals, and industrial registry systems. Bold type, minimal color, high contrast borders, and raw data presentation.

Red #C4291E accent/CTA  Dark #111111 text/headers  Bg #F5F3EE  Paper #E8E4DD

Why: Constraining to 2 real colors (red + near-black) creates a distinctive identity. Every accent means something — red = action, dark = information.

Animations

GSAP + CSS Keyframes

Hero/entrance animations use CSS @keyframes (immune to React Strict Mode double-mount). Scroll-triggered elements use GSAP ScrollTrigger with useRef guards to prevent double-animation. MagneticCard adds 3D tilt on hover.

04 — Authentication

Auth System Flow

1. User Registers

Email + password + display name → Supabase Auth creates auth user → app inserts a row in app_user table with role='viewer'.

Why viewer? New accounts start as viewers until they set up their profile. This prevents empty ghost profiles from cluttering the makers directory.

2. First Sign-In Detection

On login, the system checks if a maker_profile exists. If not → auto-creates one, awards "Curious" badge + 10 XP, and triggers the rank engine.

Why auto-create? Reduces friction. The user doesn't need to "set up" before browsing. The profile exists immediately with sensible defaults.

3. Session Management

Supabase manages JWTs with auto-refresh. The app stores tokens in localStorage with a version tag (APP_VERSION = 2.0.1). On app update, stale tokens are wiped to prevent auth errors from schema changes.

Why version-tag tokens? When the database schema changes (new columns, renamed fields), old tokens may cause mismatched queries. Auto-wiping forces a fresh login with current schema expectations.

4. Role Sync

The app fetches the user's role from app_user on every auth state change. The AuthContext provides: user object, session, role, and all auth methods (signIn, signOut, resetPassword, refreshUser).

Why re-fetch on every auth change? Admins can change a user's role at any time. Re-fetching ensures the UI reflects current permissions without requiring a logout.

07 — Achievement System

Badge System

Type 1

Progression Badges (6)

One per rank: Curious, Tinkerer, Builder, Maker, Innovator, Lab Pro. Auto-awarded when you reach the rank. These are the "main quest" badges.

Type 2

Domain Badges (7 domains × 3 tiers = 21)

Electronics, Robotics, AI, Design, Fabrication, Bio, Interdisciplinary — each with Tier 1/2/3. Earned by completing domain-specific challenges. These are the "skill tree."

Type 3

Special / Event Badges

Custom badges for achievements, events, or community contributions. Events can auto-award a badge on registration. These are the "side quests" — collectible, optional, fun.

Store Integration

Badge-Gated Products

Store products can require a specific badge to purchase. Example: advanced tools locked behind domain T3 badges. Creates a tangible reward for badge collection.

09 — Data Architecture

Database Schema

PostgreSQL on Supabase. Row Level Security (RLS) enforces access at the database level — even if the frontend is compromised, unauthorized data access is blocked.

Users

app_user

id, auth_id, email, name, role, xp, rank, is_active. Core identity table linked to Supabase Auth.

Users

maker_profile

display_name, bio, avatar, social links (github, linkedin, x, bluesky, discord, website), mentor_domains, privacy flags.

Users

xp_event

Audit log of every XP change. user_id, amount, reason, reference_id, reference_type. Full traceability.

Projects

project

title, summary, description, domain, tier, status (draft/pending/active/rejected), github_url, duration.

Projects

project_image/video/file

Media attachments with display_order for gallery sequencing.

Projects

project_member + milestone

Team members with roles. Milestones with completion tracking.

Challenges

challenge

title, tier (1-3), domain, time_estimate, mystery flag, core_idea, mission, success_criteria.

Challenges

challenge_step/material/skill/vocab

Structured content: ordered steps, required materials, skills taught, vocabulary terms with definitions.

Challenges

challenge_completion

User submissions with evidence_url, notes. Status: pending → verified/rejected by mentor.

Events

event

title, type (build_challenge/meetup/tech_tuesday), date, location, capacity, auto_badge_id.

Events

event_registration/checkin/team

Registration tracking, check-in by mentors, team formation with lead.

Badges

badge + user_badge

badge: name, tier, domain, type, criteria. user_badge: who earned what, when, awarded_by.

Store

store_product + order

Products with optional badge requirement. Orders with line items and status tracking.

Community

comment + reaction

Polymorphic: target_type (project/challenge/event) + target_id. Reactions: like/upvote/bookmark.

Operations

equipment + inventory

Equipment with induction tracking (safety training). Inventory for consumable materials.

10 — Build Pipeline

Projects Flow

Draft

Maker creates project with title, summary, domain. Can add images, videos, files, milestones, team members. Visible only to owner.

Submit for Review

Maker clicks "Submit for Review." Status changes to pending_review. Appears in mentor/admin review queue.

Review

Mentor or admin reviews the project. They can approve (→ active) or reject (→ rejected with feedback).

Active

Project appears in public archive. Owner gets 100 XP (first approval) + 200 XP (active). May unlock Builder badge.

Why require review? Quality control. The makerspace's public portfolio should only contain properly documented, real builds — not test posts or spam.
11 — Learning Paths

Challenges Flow

Tier 1 — Explorer

Beginner

Entry-level tasks across any domain. Introduces tools, safety, basic concepts. Reward: 50 XP + domain T1 badge.

Tier 2 — Solver

Intermediate

Real problem-solving. Multiple steps, materials lists, vocabulary. Requires demonstrated skill. Reward: 150 XP + domain T2 badge.

Tier 3 — Architect

Advanced

Complex multi-step projects. Visible only to Builder rank+. Reward: 400 XP + domain T3 badge. May gate store products.

Completion Flow: Maker views challenge → clicks "Mark Complete" → uploads evidence (photo/link) + notes → submission goes to pending → Mentor verifies → on verification: XP awarded, domain badge checked, rank recalculated.

13 — Marketplace

Store Flow

Browse Products

All active products displayed. Each shows name, description, price (₹), category tag, and image. Badge-gated items appear in a locked section with blur effect.

Badge Check

Products can have required_badge_id. If the user hasn't earned that badge, the product is locked. Shows which badge is needed.

Why badge-gate products? Safety. A maker shouldn't buy advanced machining tools until they've proven domain knowledge through challenges. The badge proves competency.

Purchase

Logged-in maker clicks "Purchase" → Order created in store_order + line item in store_order_item. Status: pending. Confirmation shown immediately.

14 — The Path

Complete User Journey

From first visit to Lab Pro — this is the intended progression path that the entire platform is designed around.

1. Discovery

Visitor lands on homepage → browses projects, challenges, makers → decides to join.

2. Registration → Curious (0 XP)

Creates account → auto-assigned viewer role → first login auto-creates profile + Curious badge + 10 XP. Can browse everything but can't create.

3. Profile Setup → Tinkerer (60 XP)

Completes bio + avatar + social link = 50 XP. Total: 60 XP → Tinkerer rank. Role upgraded to maker. Can now create projects and complete challenges.

4. First Challenge → Domain Badge

Completes a Tier 1 challenge = 50 XP + domain T1 badge. Attends an event = 25 XP. Total ~135 XP.

5. First Project → Builder (250 XP)

Creates project → submits for review → mentor approves = 100 XP + Builder badge. Project goes active = 200 XP. Unlocks Tier 3 challenges and showcase slots.

6. Deep Engagement → Maker (600 XP)

More challenges (T2 = 150 XP each), more projects, more events. Accumulates domain badges across multiple areas. Can propose T3 projects.

7. Mastery → Innovator (1200 XP)

Tier 3 challenges (400 XP each), multiple active projects, event presentations. Can request mentor role.

8. Leadership → Lab Pro (2500 XP + Mentor role)

Admin grants mentor role → with 2500+ XP and 3+ active projects → Lab Pro rank. Can review projects, verify challenges, manage events. Full circle: the maker becomes a guide.

17 — Common Questions

FAQ

Why Supabase instead of a custom backend?
Supabase provides auth, database, storage, and realtime in one hosted service. For a makerspace platform with <1000 users, building a custom Express/Django backend would be over-engineering. Supabase's free tier covers the scale, and Row Level Security (RLS) handles authorization at the database layer — more secure than application-level checks alone.
Why do new users start as "viewer" instead of "maker"?
Spam prevention and quality control. If new signups immediately got maker access, bots could create projects, spam challenges, and pollute the platform. Starting as viewer means you can browse but not create. Upgrading to maker requires completing your profile — a signal of genuine intent.
Why is the XP system not just a simple level counter?
Ranks are named (Curious, Tinkerer, Builder...) instead of numbered (Level 1, 2, 3...) because names create identity. A "Builder" feels like an achievement. "Level 3" feels generic. The names also describe what you can do at that level — a Tinkerer tinkers (completes challenges), a Builder builds (creates projects).
Why are challenges structured with steps, materials, skills, and vocabulary?
This mimics how real makerspace instruction works. A laser cutting challenge needs: step-by-step instructions (steps), the acrylic sheet and settings (materials), what you'll learn (skills), and terms like "kerf" or "vector cut" (vocabulary). It's a complete learning unit, not just a task description.
Why do projects require mentor review?
The public project archive is the makerspace's portfolio. If anyone could publish without review, the quality would degrade. Mentor review ensures: (1) the project is real (not a test post), (2) it's documented well enough to be useful, (3) safety standards were followed. It's quality curation, not censorship.
Why can't I see Tier 3 challenges until I'm a Builder?
Two reasons: (1) Pedagogical — advanced challenges assume knowledge from T1/T2. Showing them to beginners creates confusion and discouragement. (2) Motivational — unlocking new content feels rewarding. It gives Builders something new to discover, reinforcing the progression loop.
Why are store products badge-gated?
Safety. In a physical makerspace, you don't hand someone a CNC mill until they've completed safety training. Badge-gating is the digital equivalent: the domain badge proves you completed the relevant challenges, which means you have the knowledge to safely use the tool.
Why GSAP instead of Framer Motion or CSS animations?
GSAP's ScrollTrigger plugin is the industry standard for scroll-based animations. Framer Motion is great for component transitions but weaker for scroll-driven sequences. CSS animations can't do scroll-triggered stagger effects. GSAP gives precise control over timing, easing, and triggering.
Why CSS keyframes for hero animations instead of GSAP?
React 19 Strict Mode double-mounts components in development. GSAP's fromTo() with ctx.revert() cleanup breaks — revert sets opacity to 0, second mount animates 0→0 (invisible). CSS @keyframes are immune because they're declarative, not imperative. GSAP is still used for scroll-triggered elements with a useRef guard to prevent double-animation.
How does the polymorphic comment/reaction system work?
Comments and reactions use target_type + target_id instead of separate tables per entity. So one "comment" table handles project comments, challenge comments, and event comments. target_type = "project" + target_id = project's UUID. This avoids table proliferation and keeps the code DRY — one useComments() hook works for everything.
Why is Lab Pro rank restricted to mentors?
Lab Pro represents mastery + leadership. Having 2500 XP proves deep engagement. Having the mentor role proves the community trusts you to guide others. Both are required because XP alone doesn't prove teaching ability, and mentor role alone doesn't prove hands-on experience.
What happens if a user's role is changed while they're logged in?
The auth context re-fetches the user's role from app_user on every auth state change. So if an admin promotes a viewer to maker, the user's next action will trigger a role sync. No logout required. The ProtectedRoute component checks the live role, not a cached one.
Why version-tag auth tokens?
When the database schema changes (new columns, renamed tables), old tokens may carry stale session data that causes mismatched queries. The APP_VERSION constant (currently 2.0.1) is stored alongside the token. On app load, if the stored version doesn't match, all tokens are wiped and the user must re-login — getting a fresh session that matches the current schema.
How many Supabase storage buckets are there and why?
8 buckets: avatars, project-images, project-files, challenge-images, event-images, badge-images, product-images, equipment-images, induction-certificates. Separate buckets enable different RLS policies per content type. Avatars are public read, anyone can upload. Project files require auth. Induction certificates are mentor-only readable.
Why are all hooks in one file instead of separate files?
Discoverability. With 40+ hooks, having them in one file means Cmd+F finds anything instantly. They're grouped by entity (projects, challenges, events...) with comment headers. Splitting into 15 files would add import complexity without meaningful benefit at this scale. If the app grew to 100+ hooks, splitting would make sense.
What are the 7 domains and why these specific ones?
Electronics, Robotics, AI, Design, Fabrication, Bio, Interdisciplinary. These map to the physical zones/equipment in the Param makerspace. Electronics = soldering stations, oscilloscopes. Robotics = motors, sensors. AI = computing resources. Design = CAD workstations. Fabrication = 3D printers, laser cutters. Bio = lab equipment. Interdisciplinary = cross-domain projects.
